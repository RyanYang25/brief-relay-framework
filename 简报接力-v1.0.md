# 简报接力框架 (BRIEF Relay Framework, BRF) v1.0

> **方法定位（一句话）**：用一份持续演进的「项目简报（BRIEF）」作为唯一入口，配合「契约头 + 接力棒 + 恢复协议」三件套，让任何 AI Agent 平台都能零成本接手、跨平台可迁移、项目上下文不丢失。
>
> **名称由来**：「简报」= 核心 artifact（BRIEF.md）；「接力」= 跨 Agent / 跨平台 / 跨会话的上下文传递逻辑——谁产、谁消费、下一步动谁，显式声明，像接力棒一样一棒交一棒。
>
> **适用范围**：咨询、写作、研发、运营等一切需要多文件协作与状态跟踪的项目。领域无关，不绑定任何 IDE 或 Agent 平台。
>
> **版本说明**：本文件为首次正式命名发布版（v1.0）。其前身为内部迭代的 `project-context-prompt-v2.md`。

---

## 核心理念

### 一、BRIEF 是「入口 + 演进引擎」，不是静态门牌

BRIEF.md 同时承担两个角色：

1. **接入点**——任何新 Agent / 新会话 / 换平台时，只读这一个文件即可上手；
2. **演进引擎**——项目推进中，阶段性自动触发 + 用户手动要求时，持续更新 BRIEF 及其联动文件，让上下文随项目一起生长。

> **为什么不降级成 AGENTS.md**：AGENTS.md 是编程 Agent 圈的一个"项目级静态入口"约定——平台在新会话开始时读一次，告诉你"项目长这样"，之后它就沉默了，不具备自我演进能力。BRIEF 的本质是活的：它既管接手，也管演进。把 BRIEF 降格成 AGENTS.md，等于把一个活系统塞回静态门牌，是降维。本框架不依赖 AGENTS.md，BRIEF 自包含全部接手与演进信息。

### 二、可迁移性是第一约束

整套机制只用纯 Markdown + HTML 注释实现，**不依赖任何平台的特有能力**（无 YAML frontmatter 依赖、无平台私有记忆依赖、无"中断回调"依赖）。判定标准：把整个项目文件夹拷到另一个 Agent 平台（WorkBuddy / Trae / Qoderwork 等），新 Agent 读 BRIEF 顶部的契约头即可接手——这是本框架的硬指标。

## 提示词正文（通用搭建模板）

> 复制以下内容发给任意 Agent，替换 `{项目名称}`、`{项目描述}` 及可选占位符即可快速搭建项目基座。
> 适用于咨询、写作、研发、运营等各类需要多文件协作与状态跟踪的项目。

---

> **元指令（Guardrail）**：生成的 BRIEF.md 顶部必须包含一条防御性注释，防止 Agent 或后续协作者只修改 BRIEF.md 而忽略联动文件。参考格式：
>
> ```
> > **📌 如果你收到"更新 brief / 更新项目状态 / 同步进度"类指令**：
> > 不要只改本文件！先读本文件的「更新 SOP」章节，找到对应场景的操作指引，
> > 按 SOP 顺序联动更新 current.md / standards.md / PROJECT_GUIDANCE.md 等文件。
> ```

你是一个项目管理助手。请为项目 **{项目名称}** 搭建 project-context 工作目录。

**项目背景：** {项目描述，2-3句话说明目标、交付物、关键约束}

### 一、目录结构

```
{项目根目录}/
├── BRIEF.md                  ← 项目简报（唯一入口 + 演进引擎，放在根目录，含 @context-root 契约头）
├── {会议记录目录}/            ← 会议/沟通记录（默认 B02 会议记录/，可自定义，见下方编码说明）
├── project-context/
│   ├── context/
│   │   ├── current.md        ← 当前状态与进行中任务（更新最频繁，含 @context 契约头）
│   │   └── standards.md      ← 工作标准与输出规范（只增不改，含 @context 契约头）
│   ├── references/           ← 原始材料（数据、文档、参考资料）
│   ├── handoffs/             ← 历史产出（已完成的交付物）
│   ├── .working/             ← Agent 临时文件（自动清理，不纳入版本管理）
│   ├── changelog.md          ← 结构变更日志（仅记录文件增删、目录调整）
│   └── PROJECT_GUIDANCE.md   ← （可选）详细百科全书，记录历史决策与完整任务清单
```

**目录编码（用户个人习惯，Agent 须识别并匹配，不得强制推荐）**：用户通常用手写字母前缀对根目录文件夹分类（如 `A01 中间文档/`、`B02 会议记录/`、`C01 同事成果/`、`C02 参考资料/`）。这套编码是用户手动建立的，Agent **不应发明或推荐新编码方案**，而是：

- **首次接触项目时**：扫描根目录，识别已有的编码前缀与含义（如 `A`=过程/`B`=沟通/`C`=参考）；
- **新建或调整文件夹时**：沿用用户既有的前缀规则与编号习惯，保持连续（如新增参考类目录用 `C03 …`）；
- **在 BRIEF.md 的「文件组织」中**：如实标注用户采用的编码含义，不另行规定。

> **注意**：会议记录**不放在** project-context/ 内。建议在项目根目录下建立独立的 `{会议记录目录}/`（默认 `B02 会议记录/`），通过 BRIEF.md 的文件索引建立链接。会议记录体量大、更新频率高，独立存放避免污染 project-context 结构。

### 二、契约头规范（@context）—— 接力的握把

每个 context 文件顶部放置一个 `<!-- @context -->` HTML 注释块，声明该文件的角色与接力关系。这是「接力棒」的握把：新 Agent 一摸就知道这文件是干嘛的、什么时候该看、什么时候该改、改完下一步动谁。

**设计决策（为何用 HTML 注释而非 YAML frontmatter）**：

- YAML frontmatter 需要平台有 parser 才认；HTML 注释是任何"读文件文本"的 Agent 都能看到的通用格式，最符合「跨平台、纯 Markdown 即可接手」目标；
- HTML 注释**不渲染、不污染显示**——人类打开文件看不到这块，只有 Agent 读时才存在。

**必填字段**：

| 字段 | 含义 |
|------|------|
| `role` | 本文件在项目中的角色（如"唯一入口""活跃上下文""约束层"） |
| `produced_by` | 谁维护本文件（如"更新 SOP 自动维护""用户手动"） |
| `consumed_by` | 谁该读本文件（如"所有 Agent 每次会话开头必读"） |
| `update_trigger` | 什么条件下必须更新（如"任何状态/决策变更后"） |
| `upstream` | 上游依赖（本文件依赖谁的信息，如 BRIEF.md 依赖 current.md 的状态） |
| `downstream` | **接力棒指向**——本文件更新后，下一个该动谁（如 current.md → B01 直播课程记录/、standards.md） |

**BRIEF.md（@context-root）示例**：

```
<!-- @context-root
role: 项目唯一入口 + 演进引擎
produced_by: 更新 SOP（自动/手动触发）
consumed_by: 所有 Agent 与人类接手者，每次会话开头必读
update_trigger: 任何状态/决策/文件结构变更后
upstream: context/current.md（活跃状态）、context/standards.md（约束）
downstream: context/current.md → context/standards.md → {会议记录目录}/
-->
```

**context/current.md（@context）示例**：

```
<!-- @context
role: 当前活跃上下文（activeContext）
produced_by: 更新 SOP 场景二/三自动维护
consumed_by: 所有 Agent 每次会话开头必读
update_trigger: 任务状态变更 / 新增决策 / 发现阻塞点后
upstream: BRIEF.md（范围与约束）、standards.md（怎么做）
downstream: {会议记录目录}/、standards.md（踩坑沉淀）、PROJECT_GUIDANCE.md（重要决策）
-->
```

**context/standards.md（@context）示例**：

```
<!-- @context
role: 约束层（输出规范 / 协作流程 / 历史踩坑）
produced_by: 更新 SOP 场景五（只增不改）
consumed_by: 所有 Agent 产出前必读
update_trigger: 新增规范或踩坑记录时
upstream: BRIEF.md（项目画像与约束）
downstream: {会议记录目录}/_模板.md、references/、handoffs/
-->
```

> **接力有效性验证**：可用"全新子 Agent（不带历史记忆、禁读平台记忆目录）仅依 BRIEF + @context 块接手"的方式实测——能通过即说明项目已自描述、可迁移达标。

### 三、搭建要求

**BRIEF.md（项目简报 = 入口 + 演进引擎）**

这是整个目录的唯一入口，也是随项目演进的核心文件。任何 Agent 或人类第一次接触项目时只读这一个文件。内容控制在一页以内，包含以下板块：

1. **元指令（Guardrail）**
   
   - 在文件顶部（标题下方）插入一条防御性注释（格式见上方元指令）
   - 目的：防止 Agent 做表面更新而遗漏联动文件（如 current.md、PROJECT_GUIDANCE.md）
   - 同时标注：最后更新日期、更新人、更新原因

2. **项目画像**（稳定信息，几乎不变）
   
   - 项目名称、目标、交付物、关键约束、时间节点
   - 利益相关方：使用 ⭐ 标记决策权等级的表格格式：
     
     ```
     | 角色 | 姓名 | 决策权 |
     |------|------|--------|
     | 项目负责人 | 某某 | ⭐⭐⭐ 最高 |
     | 对接人 | 某某 | ⭐⭐ 高 |
     | 执行/顾问 | 某某 | ⭐ 中 |
     ```

3. **文件组织**（实际目录树）
   
   - 展示项目根目录的实际文件结构（不仅是 project-context/）
   - 用缩进树形展示，标注用户实际采用的编码含义（Agent 须先识别既有规则，不得套用模板预设）
   - 示例：`├── A01 中间文档/`、`├── B02 会议记录/`、`├── C01 参考资料/`（以用户实际为准）

4. **当前状态**（一句话 + 指针）
   
   - 用一句话概括当前进度，例如"第二版方案已完成，待客户反馈"
   - 指向 `context/current.md` 获取详情

5. **文件索引**（路由表）
   
   - 列出 context/、references/、handoffs/、{会议记录目录}/ 下所有文件，每个文件一行说明用途
   - 当文件增删时同步更新此索引

6. **读取规则**（渐进式披露 + 契约头说明）
   
   - 标注哪些文件必须先读（如 current.md），哪些按需读（如 references/ 下的具体文件）
   - 说明每个文件顶部 `<!-- @context -->` 块的作用，告诉接手者优先读这些块即可 30 秒掌握"项目怎么组织、每次该动哪个文件"
   - 如有 PROJECT_GUIDANCE.md，标注为"按需查阅"的第四层
   - **会话恢复协议**（见第九章）：接手可靠性不依赖"中断时自动同步"，而靠恢复协议 + 增量落盘两条规则

> **长度触发**：当 BRIEF.md 超过一页（约 80 行）时，将稳定信息之外的细节迁移到 PROJECT_GUIDANCE.md，保持入口精简。

**context/current.md（当前状态）**

记录活跃信息，更新频率最高：

- 进行中的任务列表（每条含：任务名、**状态（见下方状态枚举）**、下一步动作、阻塞点、责任人）
- 近期决策记录（**决策内容 + 依据/理由 + 日期**——"依据/理由"字段至关重要：接手者不只看结论，更要看为什么这样定，避免误改或重复讨论）
- 待解决问题
- 会议纪要索引（**只存"日期+主题+≤5条核心要点+文件路径"，不堆砌全文或长段落**）

> **长度触发**：当 current.md 超过 {N} 行（建议 120 行）时，将历史任务清单、过期决策、会议全文细节迁移到 PROJECT_GUIDANCE.md，current.md 只保留活跃信息。

**任务状态枚举（统一使用）**：
`未开始` → `进行中` → `已完成`；异常态：`已阻塞`（须注明原因）、`已取消`（须注明原因）

**context/standards.md（工作标准）**

记录输出规范和工作流程，只增不改：

- 文档格式要求（如风格、模板、命名规范）
- 质量标准（如验收条件、review要点）
- 协作流程（如角色分工、审批节点）
- 历史踩坑记录（做错过什么、怎么修正的）

**references/（原始材料）**

存放项目输入：数据文件、背景文档、客户资料、竞品分析等。BRIEF.md 的文件索引中为每个文件写一行说明。

**handoffs/（历史产出）**

存放已完成的交付物。按版本或日期命名，如 `v1-初稿.md`、`20260612-方案汇报.pptx`。当前版本不要放这里，放在项目根目录或按你习惯的位置。

**PROJECT_GUIDANCE.md（可选 —— 详细百科全书）**

当项目发展到一定规模、current.md 变得过长时启用。定位是"历史决策完整记录"，供按需查阅，不作为日常入口。

- 记录内容：完整时间节点、历史决策详情（时间+来源+内容+**理由**）、完整任务清单（含已完成和已取消）
- 与 changelog.md 的分工：
  - `changelog.md`：只记录 **文件层面的结构变更**（如"新增XX文件""删除YY目录"），供维护者看
  - `PROJECT_GUIDANCE.md`：记录 **项目层面的业务变更**（如"党组会决定XX""需求范围调整为YY"），供所有人看
- 如不需要（项目规模小），可以不加这个文件，用 current.md + changelog.md 即可

**changelog.md（变更日志）**

记录 **文件层面的结构变更**，格式：

```
## YYYY-MM-DD
- 变更内容（新增/修改/删除了哪个文件，原因）
```

> **区分**：项目业务变更（如"党组会决定XX"）记录到 `PROJECT_GUIDANCE.md` 或 `current.md`，不记入 changelog.md。changelog.md 只给维护者和 Agent 看，用于理解文件结构的演变。

### 四、渐进式披露原则

整个目录遵循四层漏斗（第四层为可选）：

- **第一层 BRIEF.md**：决定用不用、怎么用 → 快速判断是否需要深入
- **第二层 context/current.md**：决定当前该做什么 → 了解状态和任务
- **第三层 context/standards.md**：决定怎么做 → 查看规范和标准
- **第四层 PROJECT_GUIDANCE.md（可选）**：查历史决策和完整细节 → 按需查阅

每一层只回答上一层留下的问题，不要在上一层堆砌下一层的内容。

### 五、首次初始化步骤（Agent 执行顺序）

1. 创建根目录 `BRIEF.md`（含 Guardrail 注释 + **@context-root 契约头** + 项目画像骨架）
2. 创建 `project-context/` 及子目录 `context/`、`references/`、`handoffs/`、`.working/`
3. 创建 `context/current.md`、`context/standards.md`、`changelog.md`（空骨架，**每个均加 @context 契约头**）
4. 创建 `{会议记录目录}/`（如 `B02 会议记录/`）
5. 向用户确认：项目目标、交付物、关键约束、利益相关方；**识别并沿用用户已有的目录编码习惯**（如根目录已存在 `A01`/`B02`/`C01` 等前缀，直接匹配，不另起方案）
6. 回填 BRIEF.md 各板块，写入首条 changelog
7. 如项目规模预计较大，预先创建 `PROJECT_GUIDANCE.md` 骨架

> **跨会话记忆（可选）**：若 Agent 运行环境支持项目外记忆（如 `.workbuddy/memory/`），可额外维护跨会话记忆文件，记录用户偏好与项目长期约定，避免每次会话重新对齐。该文件不纳入 project-context 结构，也不影响本框架的「可迁移性第一约束」——即便没有它，仅靠项目内文件也能接手。

### 六、读取规则

当被要求处理该项目时：

1. 先读 BRIEF.md（含顶部 @context-root 块），获取全局上下文和文件索引
2. 再读 context/current.md（含 @context 块），了解当前状态
3. 根据任务需要，按需读取 context/standards.md 或 references/ 下的具体文件
4. 如需查询历史决策或完整任务记录，按需查阅 PROJECT_GUIDANCE.md（如果有）
5. 不要一次性读取所有文件

### 七、更新规则

| 文件                   | 更新频率      | 更新条件              |
| -------------------- | --------- | ----------------- |
| BRIEF.md 项目画像        | 几乎不变      | 项目目标或约束发生根本变化时才改  |
| BRIEF.md 当前状态        | 每次状态变更时   | 任务完成、新增任务、关键决策    |
| BRIEF.md 文件索引        | 每次文件增删时   | 新增或删除重要文件/目录      |
| context/current.md   | 每次任务推进时   | 任务状态变更、新增决策、发现阻塞点 |
| context/standards.md | 只增不改      | 新增规范或踩坑记录         |
| PROJECT_GUIDANCE.md  | 每次重要事件后   | 详细记录历史决策和完整任务清单   |
| changelog.md         | 每次文件结构变更时 | 新增/修改/删除文件或目录     |

> **⚠️ 注意**：以上是"该改什么"的原则表。具体操作步骤请见下方「更新 SOP」——它告诉 Agent "怎么改、按什么顺序改、哪些文件需要联动"。

---

### 八、更新 SOP（操作指引 —— 强制遵循）

> 当项目状态需要更新时，按对应场景的 SOP 操作，确保相关文件同步更新。
> Agent 收到"更新 brief / 更新项目状态"类指令时，必须先找到对应场景，再按 SOP 操作，**不要只改 BRIEF.md**。

#### 场景一：新会议纪要处理

1. **存入会议纪要**
   
   - 存入项目根目录的 `{会议记录目录}/`（默认 `B02 会议记录/`）
   - 命名规范见下方「文件命名规范」

2. **更新 current.md**（主要工作）
   
   - 读取 `context/current.md`
   - 在"会议纪要索引"中添加一条：日期 + 主题 + ≤5条核心要点 + 文件路径（**不粘贴全文**）
   - 更新"进行中任务"的状态或新增任务
   - 如有新决策，添加到"近期决策"（**记录内容 + 依据/理由 + 日期**）
   - 如有问题已解决，更新"待解决问题"

3. **同步更新 BRIEF.md**
   
   - 读取 `BRIEF.md`
   - 更新"当前状态"的一句话概括
   - 确保与 `current.md` 的"当前状态"一致
   - 如会议纪要需长期引用，在"文件索引"中添加说明

4. **如有重要决策，更新 PROJECT_GUIDANCE.md**
   
   - 读取 `PROJECT_GUIDANCE.md`（如果有）
   - 更新对应章节（时间节点、任务清单、重要决策）
   - 在"更新日志"中添加本次变更记录

#### 场景二：任务状态变更

#### 场景三：重要决策或需求变更

   - 在对应章节记录决策详情（时间、来源、内容、理由）

### 九、会话恢复协议与增量落盘纪律（跨平台接手保险）

> **平台约束（出发点）**：当前主流 Agent 平台（WorkBuddy / Trae / Qoderwork 等）在对话中断后**无法自主动作**——Agent 进程在中断时挂起，只有用户重启会话才能唤醒，它不可能在"没人唤醒"的情况下自己改写 BRIEF。因此，"会话异常结束时自动全量同步 BRIEF"这类写法在本框架下**不可实现**，必须改为"恢复时触发"的设计。

#### 9.1 会话恢复协议（触发点在"恢复时"）

任何 Agent **重新接手**本项目时（无论新会话、跨平台切换、还是中断后恢复），第一步**必须**：

1. 读取 `BRIEF.md`（含 @context-root 块）与 `context/current.md`（含 @context 块）；
2. 比对两者的最后更新时间，确认状态一致；
3. 若发现状态不一致、或存在未闭合的 in-flight 任务，先 **reconcile**（必要时向用户确认），再开始新工作；
4. 不假设"上次会话已正常收尾"——以文件内容为唯一权威。

> 该协议已通过实测：全新子 Agent（不带历史记忆、禁读平台记忆目录）仅依 BRIEF + @context 块即可正确接手并给出接力顺序，证明项目已自描述、可迁移。

#### 9.2 增量落盘纪律（根因治理）

- `BRIEF.md` / `context/current.md` 是项目**唯一权威真相源**；
- 任何任务步骤完成，**立即落盘更新对应文件**，不等会话结束统一补；
- 对话上下文（transcript）本身**不具权威性**——真正会丢的只有"还没写进文件、只活在对话里"的那部分；
- 只要每步即落盘，中断最坏只丢最后一个 in-flight 步骤，重启后顺着 @context 的 `downstream` 接力关系接着走即可。这比"自动全量同步"更可靠，因为后者还假设 Agent 活着。

#### 9.3 定时自动化巡检（可选，非必需）

WorkBuddy 等平台的 automation 按 schedule 独立跑、不依赖当前对话。可设周期性检查"BRIEF 与 current 是否一致"，但它**不知道"刚中断"**，只是机械巡检，价值有限还可能制造噪音——本框架**不将其作为主方案**，仅作锦上添花，由用户按需自定。

### 十、Agent 工作规范（强制执行）

> 以下规则针对 AI Agent，防止其污染项目结构。

1. **根目录保护**
   
   - 项目根目录只能有：BRIEF.md、project-context/、{会议记录目录}/、用户明确认可的目录
   - Agent **不得**在根目录创建临时脚本、测试文件、中间产物

2. **临时文件管理**
   
   - Agent 自生成文件（临时脚本、测试输出、中间缓存）一律放入 `project-context/.working/`
   - 任务完成后主动清理临时文件，`.working/` 不纳入版本管理
   - 如有需要保留的中间产物，先询问用户再决定存放位置（沿用用户既有的过程文档目录，如 `A01 中间文档/` 之类，不新建编码体系）

3. **文件创建原则**
   
   - 只在用户明确要求或项目 SOP 规定时创建新文件
   - 优先编辑已有文件而非新建；current.md / PROJECT_GUIDANCE.md 采用增量编辑，**避免整篇重写**

4. **增量更新纪律**
   
   - 状态更新用 Edit 定向修改对应段落，不要用 Write 覆盖整个文件
   - 整篇重写仅在文件结构重大调整或用户明确要求时进行

### 十一、文件命名规范

> 统一日期格式，避免不同协作者产生命名混乱。分隔符可用 `-` 或 `_`，但项目内须统一。

| 文件类型 | 命名格式                    | 示例                      |
| ---- | ----------------------- | ----------------------- |
| 会议纪要 | `YYYYMMDD-会议主题-纪要.md`   | `20260624-内部分工会议-纪要.md` |
| 会议原文 | `YYYYMMDD-会议主题-原文.md`   | `20260624-内部分工会议-原文.md` |
| 交付物  | `YYYYMMDD-内容描述-版本号.扩展名` | `20260630-建设方案-v3.docx` |
| 参考资料 | `YYYYMMDD-内容描述.扩展名`     | `20260515-某厅来函.pdf`     |

> 日期格式统一为 `YYYYMMDD`（8位数字），便于按时间排序和检索。会议主题尽量用短词，避免过长文件名。
